## PART 1: Introduction

Version control systems have become essential tools in modern software development, enabling teams to track code changes, coordinate collaborative work, and maintain the integrity of software projects. Among these tools, Git has emerged as the industry standard and is widely adopted in professional programming environments (Zagalsky et al., 2015). Despite its importance, a persistent gap exists between how Git is taught in undergraduate programs and the competencies graduates are expected to demonstrate in the workforce. Garousi et al. (2019) identified version control as one of the most consistently cited unmet competencies among computing graduates internationally, and this finding is reinforced by Garousi et al. (2020), who documented that the practical version control deficiencies of software engineering graduates lead employers to invest heavily in post-hire training — with consequences ranging from elevated onboarding costs to disqualification at the technical screening stage of hiring. Akdur (2022) confirmed that version control ranks among the top deficiencies flagged by industry managers evaluating new hires, a pattern that has remained stable across multiple geographic and institutional contexts.

Research consistently shows that novice developers lack a clear mental model of Git operations such as branching, merging, and conflict resolution (Isomöttönen & Cochez, 2014). Milliken et al. (2021) confirmed that this gap persists across skill levels — command familiarity does not equal command competence, and users at all experience levels routinely fail to understand how repository state changes in response to the commands they execute. Without this foundational mental model, students memorize commands without grasping how repository states change during collaboration. Embury (2019) documented that students exhibit poor Git practices and persistent misconceptions even after Git is formally introduced in coursework, and Beckman et al. (2020) found that students struggle to understand version control concepts even when Git and GitHub are integrated directly into university curricula. When conflicts arise or branches diverge, students resort to destructive behaviors: deleting and re-cloning repositories, overwriting teammates' work, or delegating all version control to a single group member (Milliken et al., 2021; Gitinabard et al., 2020). Left unaddressed, these behaviors carry directly into professional practice — graduates enter development teams unable to manage branching strategies, resolve conflicts independently, or recover from common repository errors (Guerrero-Higueras et al., 2020; Garousi et al., 2020). Wagner and Thurner (2024) argued that the field must move beyond command memorization toward conceptual understanding and practical workflows, identifying the absence of pedagogically grounded Git learning environments as a key barrier to achieving this shift.

These patterns are equally evident at Cebu Institute of Technology – University (CIT-U). A survey of thirty-five third-year BSIT students enrolled in Information Management 2 and Applications Development revealed a widespread cycle of avoidance, dependency, and destructive workarounds: 89% reported pasting Git errors directly into ChatGPT or Gemini and executing whatever command was returned without understanding it — yet only 23% had ever consulted the official Git documentation; 71% described teams that simply "picked one side" and discarded a teammate's code when merge conflicts arose; 46% reported that an AI-suggested command made their repository situation worse, and 46% also deleted the local folder and re-cloned the repository to escape a conflict; 17% abandoned the repository entirely and created a new one to resolve a conflict — representing the most extreme avoidance behavior documented in the survey. These patterns align with broader evidence that digital learning adoption barriers are particularly pronounced in Philippine higher education contexts, where locally contextualized instructional tools remain scarce (Illescas et al., 2023). When asked what they most wished a Git learning platform would teach, student responses converged on three core needs: safely resolving merge conflicts, undoing accidental commits or pushes, and practicing these scenarios step-by-step in a simulated environment before working on real projects.

The problem addressed by this study is the inability of third-year BSIT students at CIT-U to apply Git version control competently under realistic collaborative conditions — evidenced by pervasive AI dependency without comprehension (89%), destructive conflict resolution habits (71%), and avoidance behaviors ranging from re-cloning to abandoning repositories altogether. This problem is compounded by the complete absence of any locally available instructional tool that provides consequence-safe, scenario-driven practice with state-based evaluation, visual repository feedback, and difficulty-differentiated instructional support tailored to this population. The resulting skills gap has direct professional consequences: graduates who cannot independently manage branching strategies, resolve merge conflicts, or recover from destructive repository states are routinely screened out at the technical evaluation stage of software industry hiring or require extended onboarding at significant cost to their employers (Garousi et al., 2019; Garousi et al., 2020; Akdur, 2022).

Although tools such as Learn Git Branching and GitHub Classroom exist, none combine realistic repository failure states with: (1) a structured orientation unit that builds the foundational Git mental model students need before entering scenario-driven practice; (2) difficulty-differentiated instructional support that progressively reduces scaffolding as students engage with harder difficulty levels; (3) a live Directed Acyclic Graph (DAG) visualization that updates in real time after every simulator-processed command; (4) text-based repository consequence feedback that summarizes how the student's command changed the simulated repository state without labeling it correct or incorrect; (5) command-limit enforcement that discourages brute-force command entry and promotes deliberate reasoning; (6) adaptive scenario variation that serves structurally different repository templates on retry to promote genuine skill transfer; and (7) a lesson-centered content structure in which each learning unit contains lessons, and scenario practice is accessed through the lesson that teaches the relevant Git concept. GIT it! addresses this gap through a scenario-driven learning platform delivered as a browser-accessible web application.

## PART 2: Objectives

This project aims to improve Git command proficiency among third-year BSIT students at CIT-U through GIT it!, a scenario-driven web learning platform with difficulty-differentiated instructional support and realistic, state-evaluated Git practice. The proposal objectives correspond to five primary functional units: Orientation and Conceptual Readiness; Scenario Practice and State-Based Evaluation; Repository Visualization and Fading Scaffolding; Adaptive Retry and Transfer Practice; and Progress Tracking and Self-Monitoring. System authentication, account access, administrative management, AI assistance, and other support capabilities are treated as supporting or planned Phase 2 requirements because they enable or extend the platform but do not directly represent the measurable Git competency outcomes of the study. Platform performance is measured primarily through system-generated session and step logs, producing six primary KPIs and one secondary engagement metric. No pre-test, post-test, or quiz instrument contributes to primary performance measurement.

## PART 3: Methods

### System Design Overview

GIT it! is a web-based, scenario-driven learning platform delivered as a browser-accessible web application, designed to address the limited practical understanding of Git version control among third-year BSIT students at CIT-U. Rather than presenting Git commands in isolation, the platform immerses students in realistic repository problem scenarios that mirror the collaborative project conditions they encounter in courses such as Information Management 2 and Applications Development. Each scenario presents a broken or problematic repository state — such as a wrong-branch push, a merge conflict, or an accidental hard reset — and requires the student to enter Git commands that transform the repository into the expected target state.

The platform is implemented as a browser-accessible web application with a separated frontend client and backend application API. The backend manages student accounts and session data in compliance with the Philippine Data Privacy Act of 2012 (RA 10173) and applicable CIT-U institutional data privacy policies. Repository states are handled internally by the backend Repository State Simulator inside isolated server-side sessions. No external Git command-line binaries are executed, no shell commands are invoked from user input, and no connection to external Git hosting services is made at any point.

Student input passes through a command adapter controlled by the application; the adapter maps only the implemented simulator operation set to internal simulator operations. The simulator determines whether a command can be modeled by the platform, and the State-Based Evaluator separately determines whether the resulting repository state matches the scenario target. Rather than checking the specific command a student typed, the evaluator checks whether the resulting repository state matches the expected target state for that scenario step — accepting any valid command or sequence that reaches the correct final state and eliminating false negatives caused by command-string matching.

The system is governed by a No-Answer Policy: the correct command and correct command sequence are never revealed to the student at any point during or after a scenario session, at any difficulty level, on any student-facing interface surface. Expected-state diagrams on Easy and Medium show the repository goal state but never reveal the command path needed to reach it. The policy applies to all system components without exception.

### Development Methodology

GIT it! will be developed following the Agile software development methodology, which emphasizes iterative and incremental delivery, continuous feedback integration, and adaptive planning across short development cycles known as sprints (Dingsøyr et al., 2012). Agile is particularly well-suited to this project because the platform's core components — the Repository State Simulator, the State-Based Evaluator, the contextual feedback panel, and the difficulty-differentiated scaffolding structure — can be developed and tested independently across successive sprints before integration into a cohesive release. Each sprint produces a testable increment, allowing the team to incorporate feedback from early user testing and refine scenario content, feedback language, and interface design before final deployment.

### Evaluation Approach

#### Primary Performance Evaluation — System-Generated KPIs

Platform effectiveness is measured through six system-generated primary KPIs and one secondary metric derived exclusively from session and step logs, with no reliance on pre-test or post-test instruments. The Orientation Completion Gate primarily measures the Orientation and Conceptual Readiness objective. Scenario Completion Rate, Average Retry Count, Command Accuracy Rate, and Review Mode SCR primarily measure the Scenario Practice and State-Based Evaluation objective. Hard-Level Completion Rate primarily measures the Repository Visualization and Fading Scaffolding objective. Retry Transfer Accuracy primarily measures the Adaptive Retry and Transfer Practice objective. Scenario Abandonment Rate and dashboard log-display match accuracy serve as the primary indicators for the Progress Tracking and Self-Monitoring objective.

#### Usability and Technology Acceptance — External Validation

Platform usability will be assessed using the System Usability Scale (SUS), administered after the student's first completed scenario session, with a benchmark score of ≥ 70 (Brooke, 1996; Bangor et al., 2008). Student technology acceptance will be measured through a TAM-based survey capturing Perceived Usefulness (≥ 4.0), Perceived Ease of Use (≥ 3.5), and Behavioral Intention to Use (≥ 4.0) on a five-point Likert scale (Davis, 1989; Teo, 2009). SUS and TAM serve as external validation instruments supplementing the primary system-log KPI set and do not contribute to primary performance measurement.

## PART 4: Expected System

### Minimum Viable Product (MVP)

GIT it! is a web-based, scenario-driven Git learning platform designed to provide third-year BSIT students at CIT-U with a consequence-safe environment for practicing realistic version control scenarios with structured, difficulty-differentiated instructional support. The platform is organized around a curated starter content library covering foundational Git concepts through a dedicated orientation unit and a set of scenario-driven lessons. The exact count, naming, ordering, and syllabus coverage of the starter library are finalized before deployment according to approved pilot needs.

The student-facing content hierarchy is structured as follows:

```text
Learning Unit

└── Lesson

    ├── Lesson Content

    └── Scenario Skill Focus (present in scenario-bearing lessons only)

        ├── Easy difficulty instance

        ├── Medium difficulty instance

        └── Hard difficulty instance
```

- **A Learning Unit:** a top-level content group that organizes the Git syllabus for students in the unit page.
- **A Lesson:** an instructional unit within a learning unit. It contains the concept explanations, examples, command references, diagrams, and other learning materials a student needs before practice. A lesson may or may not include scenario practice — lessons without scenario skill focuses present their content directly, while lessons with scenario skill focuses attach one or more scenarios to their content.
- **A Scenario Skill Focus:** a reusable Git problem pattern attached to a scenario-bearing lesson. Examples include first commit, wrong-branch commit, partial staging, detached HEAD recovery, branch cleanup, merge conflict resolution, and undo/recovery patterns. Each scenario skill focus provides three difficulty instances: Easy, Medium, and Hard.
- **A Difficulty Instance:** the actual playable practice level for a scenario skill focus. Easy, Medium, and Hard target the same Git concept but differ in available scaffolding, command limits, and required independence.
- **A Retry Variant:** a structurally changed version of a scenario difficulty instance used after a Failed or Abandoned session. Variants alter branch names, commit structures, file-state markers, or starting repository topology so students cannot rely on memorized command sequences.
- **Unit 1: Orientation:** the first learning unit in the platform. Its lessons contain no scenario practice. Unit 1 establishes the foundational Git mental model students need before entering scenario-driven content and orients students to the platform's interface and conventions. Completion of all Unit 1 lessons is enforced as a prerequisite gate before a student may start their first scenario session, tracked by the Orientation Completion Gate (OCG). Unit 1 does not generate scenario session logs or scenario KPI data.

### Scaffolding Components

Three interface components support student reasoning within the Scenario Practice Workspace, with availability determined by difficulty level:
- **Live Animated DAG:** Updates in real time after every simulator-processed command at all difficulty levels, rendering branch pointers, commit history, HEAD position, and parent-child commit edges. Students observe the direct consequence of each command on repository state, supporting mental model development (Sorva et al., 2013). The DAG animation completes within 200 ms of each command response.
- **Expected-State Diagram:** A static diagram showing the target repository state the student must reach to complete the current scenario step; present on Easy and Medium levels only. Unlike the live DAG — which reflects the current, evolving state — the expected-state diagram is fixed and represents the goal. It is absent at Hard level, requiring students to infer the correct target from narrative context and the live DAG alone.
- **Contextual Feedback Panel:** Available only on Easy-level scenarios. After a command is processed, the panel briefly summarizes the repository-state consequence of the student's action in plain language — for example, that HEAD moved to a detached commit, a branch pointer advanced, a merge conflict appeared, or the working tree became clean. It does not label commands as right or wrong, suggest the next command, or reveal the command path. Its role is to reduce beginner cognitive load by narrating the same repository transition shown visually in the live DAG, covering non-graph details such as staging-area and working-tree state that the DAG alone does not surface (Van der Kleij et al., 2015).

### Difficulty-Differentiated Scaffolding Model

A central design principle of GIT it! is the progressive reduction of available scaffolding as students advance through harder difficulty levels within each scenario skill focus (Sweller et al., 2011). Scenario skill focuses are freely selectable at any time, but difficulty levels within a selected scenario follow a completion-based progression: Easy is available by default, Medium unlocks after Easy completion, and Hard unlocks after Medium completion. Each scenario offers three difficulty levels that target the same Git concept but progressively reduce the instructional support available:
| Level | Scaffolding | Available Components | Cognitive Goal |
|---|---|---|---|
| Easy | Full | Live animated DAG + Expected-state diagram + Contextual feedback panel + Remaining Counted-Command Counter | Supported interpretation — student connects commands to visible repository consequences with feedback describing what changed |
| Medium | Partial | Live animated DAG + Expected-state diagram + Remaining Counted-Command Counter | Visually guided reasoning — student reasons toward a fixed visual target without consequence feedback |
| Hard | Minimal | Live animated DAG + Narrative context + Remaining Counted-Command Counter | Independent command reasoning — student resolves realistic Git problems with only the live DAG and scenario narrative |

The live animated DAG remains present at all levels as the single persistent visual anchor, ensuring that students always have direct feedback on how their commands change repository state regardless of difficulty.

### Command-Count Policy and CAR Computation

Each scenario difficulty instance includes an authored command-count policy consisting of a minimum counted-command threshold, a maximum counted-command limit, and non-counted diagnostic command patterns.

The minimum counted-command threshold represents the shortest valid counted action-command path to the target repository state for that difficulty instance and is used for Command Accuracy Rate computation. A completed primary attempt at or below this threshold is counted as command-accurate for CAR. A completed primary attempt above this threshold is still accepted as a successful completion if the target repository state is reached, but it is not counted as command-accurate for CAR.

The maximum counted-command limit represents the maximum number of counted action commands allowed in a session before the session is recorded as Failed with failure reason `COUNTED_COMMAND_LIMIT_REACHED`. The Remaining Counted-Command Counter is visible to the student throughout the session and decreases only for counted action commands. Simulator-processed read-only diagnostic commands that match the authored non-counted diagnostic command patterns are logged but excluded from CAR and remaining counted-command calculations. A separate submission-safety control handles repeated diagnostic, malformed, invalid, or unprocessable submissions so that inspection commands can support good Git practice without allowing session abuse.

CAR is computed from the latest completed primary attempt per student, scenario skill focus, and difficulty level during the evaluation period. Failed attempts, abandoned attempts, and Review Mode attempts are excluded from the primary CAR denominator. This preserves the system's trial-and-error learning design: students may fail, retry, and learn through consequence-safe practice, while CAR measures whether their latest completed primary attempt for a level demonstrates command efficiency consistent with mastery.

| Attempt Type | Included in Primary CAR? | Treatment |
|---|---|---|
| Failed attempt | No | Captured by SCR, ARC, failure reasons, and retry logs |
| Abandoned attempt | No | Captured by SAR and adaptive retry logs |
| Completed primary attempt above the minimum counted-command threshold | Yes | Counts as a successful completion, but not as command-accurate for CAR |
| Completed primary attempt at or below the minimum counted-command threshold | Yes | Counts as a successful completion and as command-accurate for CAR |
| Simulator-processed command matching an authored non-counted diagnostic pattern | No | Logged for audit and state review, but excluded from CAR and remaining counted-command calculations |
| Counted action-command total reaches the maximum counted-command limit before completion | No | Session is recorded as Failed with failure reason `COUNTED_COMMAND_LIMIT_REACHED` |
| Review Mode attempt | No | Reported separately under Review Mode completion and re-engagement indicators |

### Adaptive Retry and Scenario Variation

When a student retries after a Failed or Abandoned session, the system serves a structurally different variant from the scenario's predefined pool rather than repeating the identical scenario. Variants alter branch names, commit structures, file-state markers, or starting repository topology. No Failed or Abandoned session followed by a retry results in the identical scenario being repeated when a structurally different valid variant exists. Variant IDs are logged for every attempt, enabling the system to verify that Retry Transfer Accuracy is computed only from changed-variant encounters rather than repeated identical templates.

### High-Level System Workflow

Students register and log in, then access the dashboard and unit page. The unit page displays all student-facing learning units, beginning with Unit 1: Orientation. Students must complete all Unit 1 lessons before they can start their first scenario session — this gate is enforced by the system and logged as the Orientation Completion Gate. Unit 1 lessons build the foundational Git mental model and introduce the platform's interface conventions students need before entering scenario-driven practice.

For lessons that contain no scenario practice, the student reads the lesson content and proceeds. For scenario-bearing lessons, the student reads the lesson content, clicks View Scenarios, selects a scenario skill focus, and chooses a difficulty level. Easy is available first; Medium unlocks after Easy completion for the same scenario skill focus, and Hard unlocks after Medium completion.

Each scenario difficulty instance presents a professional narrative, a simulated repository state, a terminal-style command input interface, and a live animated DAG. The scaffolding available depends on difficulty level: Easy includes the live DAG, expected-state diagram, and contextual feedback panel; Medium includes the live DAG and expected-state diagram; Hard includes only the live DAG and narrative context. The Remaining Counted-Command Counter is visible at all levels.

When a command is submitted, the simulator processes it against the internal repository state and the State-Based Evaluator checks whether the resulting state satisfies the scenario's target. Any valid command or sequence reaching the correct state is accepted. When the target state is reached, the scenario difficulty instance is marked complete, the next difficulty level is unlocked when applicable, and the attempt is logged for KPI computation. Upon successful completion, the Review Mode action becomes available for that completed difficulty instance.

If a session ends without completion because the command limit is exhausted, it is marked Failed. If a student leaves or times out without explicitly completing the session, the session is marked Abandoned. Both Failed and Abandoned sessions trigger the adaptive retry logic: the system serves a structurally different variant on the next attempt when one is available. Review Mode is not available for sessions that ended as Failed or Abandoned; those outcomes lead to a retry attempt.

Review Mode is accessed from a completed scenario difficulty instance in the dashboard, unit page, or completion screen. It reuses the Scenario Practice Workspace and allows the student to attempt the completed difficulty instance again for reinforcement. Review Mode attempts are recorded separately and do not overwrite the original completion record, unlock additional levels, award first-attempt stars, update official retry counts, or alter primary KPI computation.

All session and step interactions are logged continuously for KPI computation, including Orientation Completion Gate timestamps, started sessions, completed sessions, failed sessions, abandoned sessions, retry counts, variant IDs, command counts, failure reasons, and review-practice activity. Unit 1 lessons do not generate scenario session logs.

| Content Element | Purpose |

| Lesson | Presents instructional content; may include scenario skill focuses or may be content-only |

### Planned Phase 2 — Not Included in Current MVP Evaluation

The current semester deliverable focuses on the student-facing scenario practice platform using a static, pre-authored starter syllabus. Administrative Management and AI-Assisted Learning Support are documented as planned Phase 2 units, excluded from current-release MVP evaluation and KPI computation.

Administrative Management consolidates all non-student platform capabilities: viewing cohort-level and student-level KPI summaries, managing student accounts and access, creating and maintaining student-facing learning units, lessons, lesson content, scenario records, scenario variants, command-count policy configurations, and reviewing administrative audit logs.

AI-Assisted Learning Support includes three planned capabilities: a conceptual Git chatbot for student-facing lesson pages, an AI-assisted lesson content drafting workflow, and an AI-assisted scenario-drafting workflow. AI-generated content does not publish directly; the administrator must validate, edit, and publish all AI-generated content before students can access it.

## PART 5: Discussion

### Scope

GIT it! is scoped to third-year BSIT students at CIT-U enrolled in team-based development courses. The platform is organized around a static curated starter content library composed of learning units, lessons, scenario skill focuses, difficulty instances, and retry variants. Unit 1: Orientation provides foundational Git and platform-orientation content through eight non-scenario lessons, and its completion is enforced as a prerequisite gate before students may access scenario-driven practice. All published scenario skill focuses beyond Unit 1 are freely selectable without global unit prerequisites; difficulty levels within each scenario are completion-gated according to the Easy → Medium → Hard progression. The platform is accessible as a browser-based web application on desktop, tablet, and mobile form factors. Repository states are handled internally by the backend Repository State Simulator and are never connected to live external repositories or real Git binaries. The system supports a minimum of 40 concurrent users during the evaluation period with headroom to 100. The evaluation period is bounded by a single academic semester.

### Limitations

This study is bounded by several intentional scope decisions that constrain the generalizability of its findings. The platform is deployed at a single institution — CIT-U — targeting a specific population of third-year BSIT students, which means findings are not claimed to generalize to other programs or universities without further validation. The preliminary survey used to inform scenario selection was conducted with thirty-five students, which, while sufficient to identify the dominant failure patterns motivating the system design, may not capture the full range of Git-related difficulties across the year level. As Illescas et al. (2023) document, Philippine-specific barriers including connectivity inequality and uneven institutional digital readiness affect technology adoption in ways that differ from Western contexts, which means care must be taken when comparing outcomes against benchmarks derived from non-Philippine populations.

The platform evaluation is conducted within a single academic semester, which may be insufficient to capture long-term retention of Git competencies. The study is designed for initial skill acquisition and consequence-safe practice rather than longitudinal mastery tracking. While the simulated repository environment reflects realistic conditions, it cannot fully replicate the complexity and unpredictability of live collaborative repositories encountered in actual industry or capstone project settings. The scenario content, lesson pages, expected-state diagrams, feedback messages, and command-count policy configurations are developed by the research team based on literature and survey data.

The Orientation Completion Gate establishes that students entered scenario practice after completing Unit 1, but does not constitute a direct measure of conceptual retention from orientation content — the relationship between orientation completion and reduced retry burden is treated as an indirect indicator rather than a causally isolated finding. Command Accuracy Rate should be interpreted as a difficulty-level command efficiency indicator rather than the sole measure of Git competence. Alternative valid command paths are accepted by the State-Based Evaluator if they reach the correct repository state, even when they exceed the authored minimum counted-command threshold. The study does not attempt to isolate the independent learning effect of individual interface components — the live DAG, the expected-state diagram, and the contextual feedback panel are evaluated as a designed system rather than as independently controlled variables. The study does not separately isolate the independent effect of dashboard visibility on student learning outcomes.

### Expected Contribution

GIT it! addresses a gap that existing tools — including Learn Git Branching and GitHub Classroom — do not fill: a structured orientation unit that builds foundational Git mental models before scenario practice begins, followed by consequence-safe, scenario-driven practice with targeted lesson-level concept preparation, progressively reduced instructional support, adaptive variation for authentic transfer, and a state-based evaluator that accepts any valid approach reaching the correct repository state.

Five functional units work in sequence: Unit 1 orientation gives students the conceptual foundation and command vocabulary — including an explicit understanding of Git command anatomy and DAG literacy — they need before entering the terminal; the Lesson Page in each scenario-bearing lesson reinforces the specific concept relevant to the scenario at hand; the live DAG gives students real-time visibility into how commands change repository state at every difficulty level (Sorva et al., 2013); the expected-state diagram gives students a visual target to reason toward on Easy and Medium; and the contextual feedback panel on Easy translates repository consequences into plain language while the student's mental model is still forming (Van der Kleij et al., 2015). As instructional support decreases across difficulty levels, students transition from supported interpretation to independent reasoning — arriving at Hard-level scenarios equipped to resolve realistic Git problems with only the live DAG and narrative context (Sweller et al., 2011).

For students, the expected contribution is the ability to enter scenario practice conceptually prepared, reach Hard-level scenario completion, and demonstrate command execution efficiency consistent with difficulty-level mastery through Command Accuracy Rate — before those failures occur in graded projects. At the institutional level, GIT it! offers a replicable, evidence-grounded model for bridging the gap between introductory Git coverage in team-based development courses and the collaborative version control demands students face in capstone and industry-adjacent projects.
